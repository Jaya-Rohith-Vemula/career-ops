### Stripe's "392 new" despite an unchanged total

Stripe's active count matches the last snapshot exactly (458), but most of the underlying job ids
differ from before. Investigated: job ids are a deterministic hash of `title|url`, and the URLs
carry Stripe's own numeric requisition ids (stable, no tracking params) — so this isn't a bug in
how we build ids. Most likely explanation is real turnover on Stripe's live site between runs
(closed postings replaced by new ones) landing on a similar total by coincidence. Not something
this session's changes caused — worth a second look if it recurs on a same-day re-run.

## This session: fixed Airbnb pagination (10 → 222 jobs), generalized to all DOM-category sites

Airbnb's careers page (`careers.airbnb.com/positions/`) uses a FacetWP-style pager — no `href` at
all, just a `data-page` attribute wired to a click handler that triggers an AJAX refresh. The
pager markup doesn't even exist in the raw static HTML; it's templated in by JS from an embedded
JSON blob. The existing pagination-follow logic only understood plain `<a href>` "next" links, so
Airbnb silently stopped at page 1 (10 of 223 jobs) every run.

Fixed generically (not Airbnb-specific), in `utils/pagination.js` + `discovery/detectCategory.js`
+ `fetchers/domFetcher.js`:

- **Discovery time**: if the static HTML shows *zero* pagination signal at all, render once via
  Playwright and re-check there. If pagination only shows up post-render, the company is marked
  `requiresJs = 1` even though page 1 itself fetched fine statically.
- **Fetch time**: the static fetcher self-heals if it hits a click-only control mid-run — it
  escalates to a full Playwright re-fetch and persists `requiresJs = 1` so tomorrow's run skips
  straight to it.
- **Playwright path**: clicks visible/enabled "next"-shaped controls with no href, waits for the
  page's own listing content (not just `networkidle`, which can resolve before an AJAX pager even
  fires its request) to actually change, plus a short settle buffer since some AJAX pagers patch
  their container progressively rather than atomically.
- **Real stopping condition, not a page-count guess**: no site tells us how many pages exist.
  Pagination now stops when a page contributes zero job ids not already seen this run — that's
  what "last page" actually means, whether it's a genuinely short final page, a duplicate
  re-render, or a pager looping back to page 1. A few consecutive stale pages are tolerated
  (`STALE_PAGE_LIMIT = 3`) since an occasional click that doesn't fully land looks identical, for
  one page, to having reached the end. There's still a `MAX_PAGES` circuit breaker (200) but it's
  not expected to bind — Airbnb's real last page (23) was reached well under it.

Regression-tested against all other companies before and after — Verkada, Notion, Vercel,
Anthropic, PlanetScale, Figma all unchanged (same counts, same categories); Stripe's static
href-based pagination (unaffected by the click-path changes) still lands on 458.

## Known open issues carried over

1. **Keyword tagging substring bug (not yet fixed)** — `extractTags()` does plain substring
   matching with no word-boundary check (e.g. `react` false-matches inside `reactable`, seen in
   some Verkada descriptions pasted from Google Docs). Flagged repeatedly; still not fixed —
   deliberately deferred pending a decision, not forgotten.
2. ~~**DOM-category jobs have no `description`**~~ — **Fixed.** Each DOM-scraped job already
   carries a link to its own detail page; `domFetcher.js` now fetches that page (plain
   `fetch`+cheerio, concurrency-limited to 5 at a time) and extracts its text as `description`.
   Tested against both DOM companies: Stripe 458/458 jobs got a description, Airbnb 206/222 (the
   rest presumably 404/timeout on individual postings — not investigated further). Job detail
   pages rendered statically even for Airbnb, whose *listing* page needs JS — no Playwright
   needed for this step. Descriptions are cached per `jobId` (`getJobDescriptions()` in
   `db/client.js`) so re-runs only fetch detail pages for genuinely new postings, not the whole
   list every day.
3. **Retool / Linear still unresolved.** Retool: `careers.retool.com` has no ATS/CTA/DOM signal
   our detector recognizes (site may have changed). Linear: `linear.com` belongs to a different
   company (Analog Devices) — the real domain is `linear.app`, needs manual
   `--url --rediscover`.
4. Cron setup (Step 8 in the brief) — not yet configured.
